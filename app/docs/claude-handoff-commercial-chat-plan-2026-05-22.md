## 1. 목적

이 문서는 다음 에이전트가 대화 맥락을 다시 복원하지 않고 바로 이어서 설계와 구현을 진행하도록 만든 인수인계 문서다.

현재 목표는 단순한 상권 검색창이 아니라 아래 경험을 만드는 것이다.

> 사용자가 자연어로 지역과 업종을 물으면, 챗봇이 짧고 친숙하게 해석하고, 지도와 비교표와 그래프가 질문에 맞게 함께 바뀌는 `대화형 상권 탐색기`.

## 2. 사용자의 핵심 요구

1. 첫 화면은 질문이 중심이어야 한다.
2. 사용자가 지역을 말하면 해당 행정동 중심 지도가 열려야 한다.
3. 업종까지 말하면 점포 좌표를 찍고 점포명도 보여줘야 한다.
4. 행정동 현황 질문은 해당 동 전체의 업소, 유동인구, 업종 구성 등을 보여줘야 한다.
5. 소분류 업종 질문은 전체 업종과 무작정 비교하지 말고 같은 중분류 안의 비교를 우선해야 한다.
6. 그래프가 현재 너무 단순하다. 시계열, 비교 분석, 분포, 유동인구 패턴이 필요하다.
7. 답변이 너무 짧고 기계적이면 안 된다. 다만 루즈한 말도 싫다.
8. 답변 본문은 짧게 판단이 서야 한다. 보통 `판단 -> 근거 -> 해석` 3-5문장.
9. 후속 질문은 앞 대화 문맥을 이어야 한다.
10. `다른 동과 비교해줘` 같은 후속 질문은 2지역 비교 지도, 비교표, 비교 그래프로 전환돼야 한다.
11. 질의가 직전 답 해석 요청인지, 새 분석 요청인지, 지도/비교 도구 호출 요청인지 오케스트레이션에서 먼저 분기해야 한다.
12. Gemini 2.5 Flash 무료 API를 기본 모델로 보고 10초 안에 핵심 답을 보여주는 품질 설계를 해야 한다.
13. Gemini를 별도 학습시키는 전제는 두지 않는다. 지식은 로컬 정책과 검색형 지식배움터로 관리한다.

## 4. 제품 방향

### 4.1 제품 정체성

`범용 챗봇`도 아니고 `정적 대시보드`도 아니다.

추천 정체성:

```text
대화형 상권 탐색기
```

대화가 질문을 열고, 지도와 그래프가 답변 근거를 만질 수 있게 해야 한다.

### 4.2 첫 화면

- 질문 입력창이 중심.
- 시작 질문 칩 3-4개.
- 질문 전 지도는 숨김, 미니맵, 또는 약한 미리보기.
- 첫 화면부터 큰 지도가 질문보다 강하면 안 된다.

예시 질문:

- `중앙동은 요즘 어떤 업종이 강해?`
- `노은동 카페는 언제 붐벼?`
- `둔산1동 편의점은 어디에 몰려 있어?`

### 4.3 지역 질문

`중앙동 어때?`라면 특정 업종으로 좁히지 말고 행정동 전체 현황을 보여준다.

필수 블록:

- 기준월, 행정동, 시군구, 직접값/합성값 상태
- 업소 수
- 유동인구
- 매출 상위 업종
- 업종 구성
- 최근 변화 업종
- 동 경계 지도
- 전체 점포 분포 또는 밀도

### 4.4 지역 + 업종 질문

`중앙동 카페 어때?`라면:

- 카페 현재값
- 최근 변화
- 같은 중분류 비교표
- 카페 점포 마커
- 월별 시계열
- 점포별 합성 매출 분포는 합성임을 명시

## 5. 답변 품질

### 5.1 말투

목표:

```text
공공기관 보고서가 아니라 친숙한 상권 답변서
```

피할 것:

- `조회 결과입니다.`
- `살펴볼 필요가 있습니다.`
- `확인해 보겠습니다.`
- `읽는 편이 좋습니다.`
- 데이터가 있는데 분석 방법 설명으로 시간을 끄는 문장

### 5.2 본문 골격

본문은 보통 3-5문장.

| 문장 | 역할 |
| --- | --- |
| 1 | 질문에 바로 답하는 판단 |
| 2 | 핵심 수치 1-3개 |
| 3 | 수치 관계를 읽는 짧은 해석 |
| 선택 | 합성값/데이터 주의 |

예:

```text
중앙동 카페는 매출은 높은 편이지만 최근 한 달은 주춤했습니다.
2026년 2월 월평균 매출은 1,230만원이고 전월 대비 3.2% 줄었습니다.
업소 수가 비슷한데 매출만 빠졌다면 수요 흐름 변화를 먼저 볼 만합니다.
```

숫자 낭독만 하지 말고 관계 하나를 읽어야 한다.

- 매출과 업소 수가 같이 늘었는가
- 유동인구와 매출이 엇갈리는가
- 질문 업종이 같은 중분류 안에서 튀는가

현재 그래프는 단순 현재값과 기본 막대 수준이다. 사용자는 시계열과 비교 분석이 부족하다고 지적했다.

   - 선택 행정동
   - 시군구 평균
   - 대전 평균
   - 비교 동
3. 중분류 비교표와 비교 막대
4. 점포별 합성 매출 분포
   - box plot, strip plot, histogram 중 하나
5. 요일 x 시간대 유동인구 heatmap
6. 유사 상권 차이 막대

## 12. 데이터 파이프라인 현재 상태

### 12.1 점포 좌표

공공데이터포털 상가업소 API 수집 경로가 만들어져 있다.

관련 파일:

- `raw/store-api`

| 좌표 유효 점포 | 78,607 |

대전 5개 시군구를 조회했다.

- 동구
- 중구
- 서구
- 유성구
- 대덕구

### 12.2 점포별 합성 매출

관련 파일:

- `build/compile-microdata.js`
- `data/micro/stores-daejeon.synthetic.json`
- `data/micro/microdata-manifest.json`

현재 생성:

- 기준월: `202602`
- synthetic sales groups: `11,112`
- 점포별 매출은 합성값
- 행정동 x 업종 x 월 집계 평균을 보존하도록 배분
- 직접 관측 점포 매출처럼 말하면 안 됨

### 12.3 기초단위구 유동인구

현재 로직은 준비되어 있지만 실데이터 경계가 아직 없다.

필요 자료:

```text
SGIS 자료제공
-> 기초단위구 경계(시도)
-> 2025 SHP
```

공식 자료제공 목록에서 2025년 기초단위구 경계(시도)는 SHP, 공개, 전국, 무료로 제공된다. 좌표계는 UTM-K EPSG:5179다.

다음 처리:

1. 대전 기초단위구 SHP 다운로드
2. EPSG:5179 -> WGS84 위경도 변환
3. 대전 기초단위구 GeoJSON 생성
4. 행정동과 공간 매핑
5. 행정동 유동인구 총량을 기초단위구별 합성 배분
6. `data/micro/basic-units-footfall.synthetic.geojson` 생성

주의:

- 기초단위구 경계는 공간 틀이다.
- 현재 보유 유동인구 집계값을 기초단위구에 합성 배분한다.
- 기초단위구 유동인구도 실제 관측치처럼 말하면 안 된다.

## 15. 완료 기준

아래가 되면 1차 제품 방향이 맞게 들어간 것이다.

1. `중앙동 어때?`가 행정동 전체 현황을 보여 준다.
2. `중앙동 카페 어때?`가 같은 중분류 비교와 카페 중심 지도를 보여 준다.
3. `유동인구는?`가 직전 문맥을 이어서 동작한다.
4. `왜 그렇게 봤어?`가 새 쿼리를 돌리지 않고 직전 답의 근거를 설명한다.
5. `노은동이랑 비교해줘`가 비교표와 비교 지도 모드로 간다.
6. 일반 질의 90% 이상이 10초 안에 핵심 답을 보여 준다.
7. Gemini가 실패해도 로컬 분석 카드가 비지 않는다.
8. 합성 매출과 합성 유동인구가 실제 관측치처럼 말되지 않는다.

## 16. 위험과 주의

- Gemini 무료 API 호출을 질문마다 3회 이상 기본으로 만들지 말 것.
- 모델이 계산하거나 없는 원인을 만들게 하지 말 것.
- 합성 점포 매출로 점포 랭킹을 단정하지 말 것.
- 기초단위구 합성 유동인구를 관측 유동인구처럼 말하지 말 것.
- 지역/업종이 모호하면 억지 추정하지 말고 한 번만 확인할 것.
- 사용자는 긴 문장과 느슨한 분석 방법 설명을 싫어함.
- 답변은 친숙해야 하지만 홍보 문구나 창업 성공 보장은 금지.
