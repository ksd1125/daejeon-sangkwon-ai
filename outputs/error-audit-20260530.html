B. 집계·데이터 정확성

#4 Overview "247개 업종이 영업 중" — 모든 동에서 동일 High

행정동 전체 현황의 "업종 수"가 해당 동 실제 영업 업종 수가 아니라 전체 카탈로그(247)로 표시됩니다. 대청동(전체 130업소)도, 산내동(749), 중앙동(1,984)도 모두 "247개 업종". 130개 업소가 247개 업종을 다 영업할 수 없으므로 명백한 오집계입니다. 요약 문장과 통계 카드 셀("업종 수: 247개") 양쪽에 노출.

증거
대청동·대흥동·기성동 등 6개 동 일괄 검증 → 전부 247
위치
response-builder.js:661 overview.totalIndustries||0 → 670 출력. 근본은 query-engine.js buildOverview의 totalIndustries 산출(카탈로그 길이로 추정)
수정안
해당 동에서 매출/업소 데이터가 존재하는 업종만 카운트

#5 정형외과 → "성형외과 의원" 오매칭 High

업종 부분매칭이 "정형외과"에서 "정"을 떼고 "형외과"를 "성형외과 의원"에 매칭(ratio 0.5)합니다. 정형외과(整形外科)와 성형외과(成形外科)는 전혀 다른 진료과로, 사용자가 받는 데이터가 완전히 다른 업종이 됩니다. 의미적으로 위험.

parse("중앙동 정형외과 매출") → industry=성형외과 의원, raw="형외과", type=partial
위치
intent-parser.js 부분매칭(ratio≥0.35 허용) — 의료 "○○과" 계열 substring 충돌
수정안
"○○과" 의료 토큰은 부분매칭 제외(정확/별칭만), 또는 접두 1글자 차이 페널티 강화

#6 데이터 결손 동 합산 → "총 업소 0개" + 모순된 평균 High

법정동 합산(merge) 시 구성 행정동이 모두 sgg_sub(개별 업소 데이터 없음)이면 합산 업소가 0개로 나오는데, 동시에 "평균 업소당 월매출 1,019만원"을 표시합니다. 업소 0개인데 업소당 평균 매출이 존재하는 논리적 모순.

"법동 한식 매출" → "…업소 규모는 0개로 집계했습니다. 평균 업소당 월매출 1,019만원" (총 업소 수(합) 0개)
위치

수정안
